# Obyava

An olx-style app built with JS and supabase. Users login /register, then public an advertisements, view for every advertisement, create, edit, delete an advertisement on the advertisement view

## Architecture and Technology Stack

A classic client-server app:
 - Front-end: JS app;
 - Backend: Supabase;
 - Database: PostgreSQL;
 - Authentication: Supabase Auth;
 - Build tool: Vite npm;
 - API: Supabase REST API;
 - Hosting: Netlify;
 - Source code: GitHub;

 ## Backend and Database

- Use **Supabase** as the backend and **PostgreSQL** as the database.
- Use **Supabase Auth** for user authentication and authorization.
- Use the **Supabase REST API** for all backend API calls.

- Implement **Row-Level Security (RLS)** policies to ensure proper access control:
  - Ordinary users can create, view, edit, and delete **only their own advertisements**.
  - Not logged-in users can **view only published advertisements**.
  - Admin users can access, review, approve, edit, or archive **all advertisements**.

- Use appropriate **database indexes** to optimize common operations such as:
  - searching advertisements by title,
  - filtering by category or location,
  - retrieving advertisements owned by a specific user.

- Use **database transactions** to ensure data integrity when creating, editing, or deleting advertisements, especially when multiple related operations are involved.

- Use **Supabase Storage** for storing advertisement images.
  - Implement appropriate security policies so that only authorized users can upload images.
  - Images should be referenced in the database via file paths, not stored directly in tables.

- Use a **consistent naming convention** for database tables, columns, and relationships to improve readability and maintainability.
- Manage all database changes using **Supabase migrations**.
  - Always track schema changes with migrations.
  - Keep a copy of all migration SQL files in the repository for reference and rollback.
  - Use a consistent naming convention for migration files (timestamp + short description).

### User Profile (`users`)

Stores additional application-level user data linked 1:1 with Supabase Auth.

- `auth.users` = Supabase-managed authentication table (source of truth for auth)
- `public.users` = application-level profile table
- `id` – UUID, references `auth.users.id`
- `full_name` – user display name
- `phone` – contact phone for ads
- `avatar_url` – profile picture URL
- `role` – authorization role (`user`, `admin`)


## Authentication and Authorization
 - Use Supabase Auth for user authentication and authorization.
 - Implement RLS policies to restrict access to  data based on user roles and permissions. For example, only logged-in users should be able to create, edit, or delete advertisements, and only admin users should be able to access the admin dashboard and user management features. The admin user can access and edit/delete/approve all the advertisements of all users. Use JWT tokens for authentication and authorization in API calls to the backend. Ensure that all sensitive data, such as passwords and authentication tokens, are securely stored and transmitted using encryption.
